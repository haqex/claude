# Pipeline Definition Format

A pipeline is a `.md` file in `pipelines/` with YAML frontmatter defining steps, and optional prose describing context or usage.

---

## Frontmatter Schema

```yaml
---
name: <kebab-case-name>           # matches the filename, used in /project:run-pipeline <name>
description: <what this does>     # shown in the execution plan

inputs:                           # runtime values the pipeline accepts
  - name: <key>                   # referenced as {{input.<key>}} in step args
    description: <what it is>
    required: true | false

steps:
  - id: <step-id>                 # referenced by subsequent steps as {{<id>.<field>}}
    skill: <skill-name>           # SKILL.md name inside the domain
    domain: <domain-name>         # which domain the skill lives in
    args: "<argument string>"     # passed as $ARGUMENTS to the skill; supports {{}} interpolation
    after: [<step-id>, ...]       # optional: step IDs this step depends on (for ordering/clarity)
    on_error: stop | skip | ask   # optional: what to do if this step fails (default: stop)
---
```

---

## Interpolation

Inside `args`, you can reference:

| Syntax | Resolves to |
|--------|-------------|
| `{{input.<key>}}` | A value from the pipeline's `inputs` |
| `{{<step-id>.<field>}}` | A named output from a previous step's execution context |

Named step output fields are not strictly typed — they're resolved by Claude from context. Common field names by convention:

| Field | Meaning |
|-------|---------|
| `topic` | The primary subject extracted from the step (e.g. video topic) |
| `title` | Title of the content ingested |
| `skill_name` | Name of a skill generated by the step |
| `note_path` | Vault path of a note created by the step |
| `domain` | Domain a generated artifact was placed in |

## Example

```yaml
---
name: youtube-research
description: Ingest a YouTube video — transcript → vault note → diagrams → MOC update
inputs:
  - name: url
    description: YouTube video URL
    required: true
steps:
  - id: ingest
    skill: youtube-to-skill
    domain: learn
    args: "{{input.url}}"

  - id: diagrams
    skill: generate-diagrams
    domain: obsidian
    args: "{{ingest.topic}}"
    after: [ingest]

  - id: moc
    skill: generate-moc
    domain: obsidian
    args: "YouTube"
    after: [ingest]
    on_error: skip
---
```

---

## Tips

- Steps run in the order listed. Use `after` to make dependencies explicit and readable.
- `on_error: skip` is useful for optional steps (like MOC updates) that shouldn't block the pipeline.
- `on_error: ask` is useful when a step's failure requires a human decision before continuing.
- Keep pipelines focused. A pipeline that does 8 things is harder to debug than two pipelines that do 4.
